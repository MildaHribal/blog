Launching a new project or a major feature without a checklist is a gamble. Not because you'll forget something obvious, but because launch pressure is the worst time to be thinking clearly. Having a list means you run the same checks every time, no matter how late it is.

This is mine.

## Environment and configuration

- [ ] All required env variables are set in production
- [ ] Env variables are not committed to git (`.gitignore` includes `.env`)
- [ ] Secrets are rotated from any dev values used during setup
- [ ] `NODE_ENV=production` is set
- [ ] Database connection uses the production endpoint (not dev/staging)

A surprising number of incidents are "we forgot to set `STRIPE_SECRET_KEY`" or "we pointed production at the staging DB".

## Security

- [ ] Security headers configured (CSP, X-Frame-Options, Referrer-Policy)
- [ ] HTTPS enforced, HTTP redirects to HTTPS
- [ ] Admin/internal routes protected by auth
- [ ] API routes validate input (don't trust the client)
- [ ] `robots.txt` blocks admin and API paths

```
User-agent: *
Disallow: /admin
Disallow: /api/
```

## Performance

- [ ] Images are compressed and using modern formats (WebP/AVIF)
- [ ] Static assets are served with long-lived cache headers
- [ ] Core Web Vitals measured (LCP < 2.5s, CLS < 0.1, INP < 200ms)
- [ ] Lighthouse score checked on mobile, not just desktop
- [ ] No console errors in production mode

## Monitoring

- [ ] Error tracking is active (Sentry, or even just logging to a service)
- [ ] Uptime monitoring configured (a simple ping every 5 minutes is enough to start)
- [ ] At least one alert that notifies you when the site is down

I've shipped things without monitoring and found out about outages from users. Never again.

## Database

- [ ] Migrations are applied
- [ ] Indexes are in place for common query patterns
- [ ] Backups are enabled and tested (actually restore from one)
- [ ] Connection pool size is appropriate for expected traffic

## DNS and domains

- [ ] DNS TTL is set appropriately (lower before a cutover, raise after)
- [ ] HTTPS certificates are provisioned and auto-renewing
- [ ] `www.` redirects to the canonical URL (or vice versa)
- [ ] Old URLs redirect correctly (301, not 302)

## One thing people skip

Actually testing the happy path in production after deploy. Not in staging, not locally — in production. Create an account, do the main user action, verify the email arrives.

Staging is not production. There are always differences.

The checklist takes 20 minutes. It has saved me from going to bed thinking everything was fine and waking up to user complaints. Worth it.
