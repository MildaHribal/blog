Most developers know `Content-Type` and `Authorization`. There's a whole tier of headers below that that most apps either set wrong or don't set at all.

## Security headers you should just have

```
X-Content-Type-Options: nosniff
```
Tells browsers not to sniff the MIME type. Prevents a class of attack where a malicious file is served as plain text and executed.

```
X-Frame-Options: DENY
```
Prevents your page from being embedded in an iframe on another site. Stops clickjacking.

```
Referrer-Policy: strict-origin-when-cross-origin
```
Controls how much of the URL is sent in the `Referer` header when navigating between pages. The default leaks more than you'd expect.

```
Permissions-Policy: camera=(), microphone=(), geolocation=()
```
Explicitly opt out of browser features you don't use. Reduces attack surface.

In Next.js, set these in `next.config.ts`:

```ts
const nextConfig = {
  async headers() {
    return [
      {
        source: "/(.*)",
        headers: [
          { key: "X-Content-Type-Options", value: "nosniff" },
          { key: "X-Frame-Options", value: "DENY" },
          { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
          { key: "Permissions-Policy", value: "camera=(), microphone=(), geolocation=()" },
        ],
      },
    ];
  },
};
```

## Cache-Control: the one people always mess up

```
Cache-Control: max-age=31536000, immutable
```

For static assets with content hashes in the filename (like `main.a3b4c5.js`), this tells browsers to cache them for a year and never revalidate. Since the filename changes when content changes, there's no stale data risk.

```
Cache-Control: no-cache
```

People confuse this with `no-store`. `no-cache` means "cache it, but always revalidate". `no-store` means "don't cache at all". For HTML pages that change, `no-cache` is usually what you want.

## Vary: the forgotten one

```
Vary: Accept-Encoding
```

If your server sends gzip or brotli, this header tells CDNs to cache separate versions for compressed and uncompressed. Without it, a CDN might serve a gzip response to a client that asked for plain text.

## Content-Security-Policy: worth the pain

CSP is the hardest one to set up and the most powerful for protection:

```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';
```

Start in report-only mode:

```
Content-Security-Policy-Report-Only: default-src 'self'; report-uri /csp-reports
```

This lets you see what would be blocked without actually breaking anything.

These headers take 30 minutes to add and protect against a long list of attacks. Few apps bother. Add them.
