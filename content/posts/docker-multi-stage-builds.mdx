I pushed my first Node.js Docker image at 1.3 GB. The application was a simple Express API. The image contained dev dependencies, the TypeScript compiler, source maps, and a lot of `node_modules` that had no business being in production.

Here's what I do now.

## The naive Dockerfile

```dockerfile
FROM node:20
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

This image includes everything: source TypeScript, all dev deps, the TS compiler itself. When your app starts, it's carrying 800MB of baggage it never uses.

## Multi-stage: build vs. runtime

```dockerfile
# --- Stage 1: Build ---
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# --- Stage 2: Runtime ---
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production

COPY package*.json ./
RUN npm ci --omit=dev

COPY --from=builder /app/dist ./dist

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

The final image only contains:
- The compiled output (`dist/`)
- Production `node_modules`
- Node runtime (Alpine, ~130MB)

My 1.3 GB image became 280MB. Same application.

## For Next.js specifically

Next.js has a built-in output mode for this:

```js
// next.config.ts
const nextConfig = {
  output: "standalone",
};
```

This generates a `/.next/standalone` folder with only the files needed to run the app.

```dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production

COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

EXPOSE 3000
CMD ["node", "server.js"]
```

The result is an image under 200MB that starts in under 2 seconds.

## Other things worth doing

**.dockerignore** — people forget this:

```
node_modules
.next
.git
*.env
*.log
```

Without it, `COPY . .` copies `node_modules` into the build context, slowing down every build.

**`npm ci` instead of `npm install`** — deterministic, faster in CI, doesn't touch `package-lock.json`.

**Pin your base image** — `node:20-alpine` can change. For reproducible builds, pin to a digest or a patch version like `node:20.12-alpine`.

Start with the right setup and you won't spend a Friday afternoon wondering why your deploys take eight minutes.
