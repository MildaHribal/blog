There's a pattern in TypeScript code that looks harmless and causes real pain:

```ts
try {
  await doSomething();
} catch (e: any) {
  console.error(e.message); // might be undefined. might crash. nobody knows.
  setError(e.message);
}
```

The `catch` clause in JavaScript can receive anything: an `Error`, a string, a number, `null`, an object with no `message` property. TypeScript knows this — that's why the default type is `unknown` since TS 4.0.

Using `any` just opts you back out of that protection.

## The right shape for error handling

```ts
function getErrorMessage(error: unknown): string {
  if (error instanceof Error) return error.message;
  if (typeof error === "string") return error;
  return "An unexpected error occurred";
}

try {
  await doSomething();
} catch (error) {
  setError(getErrorMessage(error));
}
```

This function narrows the type properly and always returns a string you can safely display.

## Typed custom errors

For cases where you need to differentiate error types:

```ts
class ApiError extends Error {
  constructor(
    public readonly status: number,
    message: string
  ) {
    super(message);
    this.name = "ApiError";
  }
}

try {
  const res = await fetch("/api/posts");
  if (!res.ok) throw new ApiError(res.status, "Request failed");
  return await res.json();
} catch (error) {
  if (error instanceof ApiError && error.status === 401) {
    redirect("/login");
  }
  throw error; // re-throw what you can't handle
}
```

Now you can catch specifically and let everything else bubble up.

## Result types instead of exceptions

For operations where failure is expected (not exceptional), a result type is cleaner than try/catch:

```ts
type Result<T, E = Error> =
  | { success: true; data: T }
  | { success: false; error: E };

async function parseConfig(raw: string): Promise<Result<Config>> {
  try {
    const data = JSON.parse(raw) as Config;
    return { success: true, data };
  } catch {
    return { success: false, error: new Error("Invalid JSON") };
  }
}

const result = await parseConfig(rawInput);
if (!result.success) {
  console.error(result.error.message);
  return;
}
// result.data is fully typed here
```

The caller can't forget to handle the error case — TypeScript enforces it.

## The rule of thumb

- Expected failures (invalid input, not found, network errors): use result types or check `instanceof`
- Unexpected failures (bugs, invariant violations): let them throw and catch at the boundary
- Never use `any` in a catch clause

The five minutes spent handling errors properly saves hours of debugging "Cannot read property 'message' of undefined" at 11pm.
